# 系统架构设计

## 1. 概述

本项目是一个**钉钉AI表格驱动的图片生成服务**，核心流程如下：

```
┌─────────────────────────────────────────────────────────────────┐
│                        钉钉AI表格（前端）                         │
│  字段：编号 | 款号 | 素材图 | 提示词 | 生图模型 |                │
│        生成图片 | 生成结果 | 生成时间 | 生成按钮                  │
│                                                                  │
│  1. 协作者填写款号、提示词、选择生图模型、上传素材图             │
│  2. 点击"生成按钮"                                               │
│  3. 钉钉自动化流程 → HTTP POST 记录ID                            │
└──────────────────────────┬──────────────────────────────────────┘
                           │ HTTP POST /api/v1/generate
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                       Python 后端服务                             │
│                                                                  │
│  ┌──────────┐    ┌──────────────┐    ┌───────────────────┐      │
│  │  API层   │───▶│  业务服务层   │───▶│  钉钉集成层       │      │
│  │ (FastAPI)│    │ (services/)  │    │  (dingtalk/)      │      │
│  └──────────┘    └──────┬───────┘    └───────────────────┘      │
│                         │                                        │
│                         ▼                                        │
│                  ┌──────────────┐                                │
│                  │  AI生图引擎   │                                │
│                  │ (generator/)  │                                │
│                  └──────────────┘                                │
│                                                                  │
│  4. 接收记录ID → 5. 钉钉SDK获取记录数据                          │
│  6. 下载素材图 → 7. 调用AI生图                                   │
│  8. 回写结果到钉钉表格（生成图片、生成结果、生成时间）            │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 技术选型

| 组件 | 技术 | 选型理由 |
|------|------|----------|
| Web 框架 | FastAPI | 异步、自动OpenAPI文档、类型安全 |
| ASGI 服务器 | Uvicorn | 轻量、高性能 |
| 钉钉 SDK | dingtalk-sdk 或直接 OpenAPI | 读写多维表格、文件上传 |
| HTTP 客户端 | httpx | 异步、支持连接池 |
| 数据验证 | Pydantic v2 | 类型校验、序列化 |
| 图片处理 | Pillow | 格式转换、尺寸调整 |
| 任务队列 | asyncio 后台任务 / Celery | 生图耗时，需异步 |
| 配置管理 | pydantic-settings | 环境变量/配置文件 |
| 日志 | loguru | 结构化日志、更友好 |
| 测试 | pytest + pytest-asyncio | 标准异步测试框架 |

## 4. 数据流详解

### 4.1 触发流程

```
用户点击"生成按钮"
  → 钉钉表格自动化规则匹配到该事件
  → 自动化动作：发送 HTTP 请求
  → POST /api/v1/generate  { "record_id": "recXXXXXX" }
```

### 4.2 后端处理流程

```
FastAPI接收请求 → 放入后台任务（立即返回 202 Accepted）
  │
  ├─ 1. 通过钉钉SDK获取记录详情
  │     GET /v1/table/{table_id}/records/{record_id}
  │     → 拿到：素材图URL、提示词、生图模型
  │
  ├─ 2. 下载素材图（如果存在）
  │     httpx.get(素材图URL) → 本地临时文件
  │
  ├─ 3. 调用AI生图引擎
  │     generator.generate(prompt, model, reference_image)
  │     → 返回生成图片的字节流
  │
  ├─ 4. 上传生成图片到钉钉云空间
  │     POST /v1/drive/spaces/{space_id}/files
  │     → 拿到生成图片的file_id/URL
  │
  └─ 5. 更新钉钉表格记录
        PUT /v1/table/{table_id}/records/{record_id}
        → 写入：生成图片URL、生成结果="成功"、生成时间
```

### 4.3 错误处理流程

```
任一步骤失败 → 更新钉钉记录"生成结果"字段为错误信息
  → "失败: {error_message}"
  → 同时写入"生成时间"标记处理时间点
```

## 5. 关键设计决策

### 5.1 为什么用后台任务而非同步返回？

AI生图通常耗时 5-60 秒，钉钉自动化回调有超时限制（通常 5 秒）。后端必须在收到请求后**立即返回**，然后异步处理。返回 202 Accepted，后续通过回写表格来通知结果。

### 5.2 为什么通过记录ID获取数据而不是请求体传全部数据？

钉钉自动化的 HTTP 请求体大小有限制。只传记录ID，后端通过SDK去拉取完整数据，更可靠、更安全。

### 5.3 生成结果如何回传？

通过钉钉多维表格API（OpenAPI）的更新记录接口，直接写入"生成图片"附件字段和"生成结果"文本字段。"生成时间"字段通过表格的公式字段自动填充，或在更新时一并写入。

## 6. 安全考虑

- **API鉴权**：需实现一个简单的 token/callback 验证，防止未授权调用
- **钉钉凭证**：AppKey/AppSecret 通过环境变量注入，不写入代码
- **文件隔离**：素材图下载后处理，避免路径穿越
- **速率限制**：对同事并发点击做控制，避免生图API超限

## 7. 部署架构（初期）

```
单机部署：
  Uvicorn (多 worker)
    ├── FastAPI 主进程
    └── asyncio 后台任务

后期可扩展为：
  FastAPI (接收请求) → Redis (消息队列) → Celery Worker (处理生图)
```
